# Implementation Status

> Generated from `docs/internal/feature-status.json` by `scripts/render-feature-status.mjs`. Do not edit this file by hand.

This is the canonical human-readable status view for the repo. Use source code, config, tests, and workflow files as primary evidence when updating the JSON inventory.

## Status vocabulary

- `implemented`: A source-backed concrete implementation exists and is reachable through exports, runtime code, or an active workflow.
- `partial`: A concrete path exists but is incomplete, latent, or only partly wired.
- `interface-only`: A contract or export exists without concrete in-repo implementation or wiring.
- `missing`: The repo currently has no concrete backing for a tracked surface implied by contracts, surrounding code, or maintained docs.

## Maintainability rules

- Track only high-value surfaces that help future engineers and agents avoid unsafe edits or false assumptions.
- Every entry must cite one to three evidence paths from code, config, tests, or workflow files.
- Consumer docs in apps/docs and README files may support a note, but they must never be the sole evidence.
- Partial and missing entries must explain the concrete gap or latent wiring.
- IMPLEMENTATION_STATUS.md and FEATURE_MATRIX.md are generated views and must stay in sync with this file.
